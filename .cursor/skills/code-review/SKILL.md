# Code Review

对指定代码进行结构化审查，按清单逐项检查并给出分级反馈。

## 何时使用

- 用户说「codereview」「code review」「代码审查」「帮我看看这段代码」「审查一下」等
- 用户贴出代码或指定文件/PR 要求审查
- 用户问「这段代码有没有问题」「能不能优化」

## 审查流程

1. **确认范围**：弄清要审查的代码（文件、片段、或 PR 变更）。
2. **按清单检查**：依次过一遍下面的审查清单。
3. **输出反馈**：按「反馈格式」整理成可执行的建议。

## 审查清单

逐项检查，未通过项在反馈中说明位置与建议修改。

### 正确性与健壮性
- [ ] 逻辑正确，边界与异常情况有考虑
- [ ] 无明显的 off-by-one、空指针、除零等错误
- [ ] 错误处理完整（捕获、日志、降级或向上抛出）
- [ ] 并发/多线程场景下无竞态或死锁风险（如适用）

### 安全
- [ ] 无 SQL 注入、XSS、命令注入等风险
- [ ] 敏感信息（密钥、密码）不写死、不打印到日志
- [ ] 输入校验与权限检查到位（若涉及用户输入或权限）

### 可读性与可维护性
- [ ] 命名清晰，函数/变量用途一目了然
- [ ] 函数长度与职责合理，避免巨型函数
- [ ] 重复逻辑已抽取，符合项目既有风格
- [ ] 关键逻辑有注释，复杂算法或业务规则有说明

### 测试与质量
- [ ] 关键路径有测试覆盖（若项目有测试）
- [ ] 修改未破坏已有行为（回归风险）
- [ ] 依赖与副作用明确，便于单测与集成

### 性能与资源（若相关）
- [ ] 无明显 N+1、大集合遍历、重复计算
- [ ] 资源（连接、文件句柄等）正确释放
- [ ] 算法/数据结构与数据规模匹配

## 反馈格式

每条反馈标明**严重程度**和**位置**，并给出具体修改建议。

- **严重**：必须修复（bug、安全漏洞、会导致线上故障）
- **建议**：建议修复（可维护性、性能、风格不统一）
- **可选**：锦上添花（命名、注释、小优化）

示例：

```markdown
### 严重
- **文件:行号** — 问题简述。建议：具体改法或示例代码。

### 建议
- **文件:行号** — 问题简述。建议：具体改法。

### 可选
- **文件:行号** — 问题简述。
```

若某类没有项，可省略该小节。最后用 1～2 句话总结整体评价和优先建议。

## 输出结构

审查结果按以下结构组织（可据实际情况精简）：

```markdown
# Code Review 报告

## 范围
[被审查的文件/片段/PR 简述]

## 审查结果
[按「反馈格式」列出的严重 / 建议 / 可选]

## 总结
[整体评价与优先建议]
```

## 注意事项

- 紧扣用户给出的代码范围，不随意扩大审查范围除非用户同意。
- 建议要可执行：指出位置并给出改法或示例，避免空泛。
- 若项目有 `.cursor/rules`、CONTRIBUTING、lint 配置等，审查时参考其中约定。
- 不确定处可标注「建议与作者确认」而非武断结论。
